# 09｜用 AI 寫 code：非工程師的 SvelteKit 實戰

## 這篇解決什麼問題

你不是工程師，但想做一個自己的網站或工具。AI 讓這件事變得可能——但過程中有很多「非工程師特有的坑」。這篇講的不是技術教學，是一個非工程師用 AI 寫 code 的真實經歷。

## 我原本的做法 → 現在的做法

**原本：** 想做一個個人知識管理系統。傳統路線：先去學 HTML → CSS → JavaScript → 框架 → 資料庫。預計三到六個月。實際上，學了一個月就放棄了，因為離「能做出東西」還太遠。

**現在：** 用 Claude 寫 SvelteKit。第一天就有一個可以跑的頁面。不懂的部分問 Claude，它一步一步帶。三個月後，BIU_AI_OS 已經是一個可用的系統——有前端、有後端、有資料庫、部署在 Vercel 上。

我現在還是不會寫 code。但我會用 AI 寫 code，然後判斷結果對不對。這是兩件完全不同的事。

## 完整流程

### 步驟一：定義你要做什麼（不是學什麼）

```
我不是工程師，但我想做一個 [描述你想做的東西]。

功能需求（越簡單越好）：
1. [功能 1]
2. [功能 2]
3. [功能 3]

請幫我：
1. 判斷這個用 SvelteKit 做合不合適
2. 如果合適，列出需要的技術元件（用白話解釋每個是什麼）
3. 給我一個最小可行版本（MVP）的範圍建議
4. 預估我需要多少時間（假設我完全靠 AI 寫 code）
```

### 步驟二：一步一步建

```
我們來開始建 [專案名稱]。

我的技術程度：非工程師，會基本電腦操作，沒寫過 code。
開發環境：[Mac/Windows]

請從最基礎開始，一步一步帶我：
1. 先告訴我要安裝什麼（給我可以直接複製貼上的指令）
2. 每一步都等我確認完成後再進下一步
3. 如果有錯誤訊息，我會貼給你，請告訴我怎麼解決
4. 每完成一個階段，告訴我「目前做到哪了」和「接下來要做什麼」
```

### 步驟三：維護的心態

AI 寫的 code 跑起來了，不代表事情結束。最重要的習慣是：

**每完成一個功能就備份一次。** 用 Git（讓 AI 教你怎麼用，五分鐘就會了）。壞了可以回退，不用從頭來。

**錯誤訊息是你最好的朋友。** 不要自己猜哪裡出問題。把完整的錯誤訊息複製貼給 AI，它通常能直接告訴你原因和修法。

我最早是用 ChatGPT 寫 code，做出來很多垃圾。後來試 Gemini，也是做了很多無謂的東西。問題不在工具——是我不知道怎麼跟 AI 協作寫 code。我以為把需求丟進去就會得到成品，結果得到的是一堆看起來能跑、實際上一塌糊塗的東西。轉折點是我開始用 Claude Code，而且學會了「一步一步來、每步都驗收」的節奏。不再期待一次到位，而是把大目標拆成小步驟，每完成一步就確認一次。BIU_AI_OS 就是這樣一步一步建起來的。

## 常見卡關點

**環境設定是最容易放棄的關卡**
安裝 Node.js、設定開發環境、跑第一個指令——這段路對非工程師來說像在走迷宮。但只要照 AI 給的指令一步一步來，通常半小時內就能搞定。撐過這關，後面會順很多。

**「AI 寫的 code 跑不起來」**
百分之九十的情況是：版本不對、套件沒裝、或者路徑寫錯。把完整錯誤訊息貼給 AI，不要只說「它壞了」。

**「改了一個地方其他地方壞了」**
這就是為什麼要用 Git。非工程師最常犯的錯是：不備份就改，改壞了回不去，然後放棄。Git 讓你「後悔」的成本幾乎為零。
